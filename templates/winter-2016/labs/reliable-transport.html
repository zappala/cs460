You will use the simulator to examine the efficiency achieved by different window sizes. ## Simulator The [Bene](https://github.com/zappala/bene) simulator includes stubs for TCP, including an implementation of TCP buffering. ## TCP Implement reliable transport using TCP as your model. The TCP code in the simulator is written so that no opening handshake is required; a host can immediately begin sending data. We will only simulate data traveling one way from the sender to the receiver, but the code is written so that you could send data in both directions. You should implement the following for the **sender**: * Add all data the application wants to send to a send buffer. Data stays in the send buffer until TCP is ready to send it and is removed once it is ACKed. * Use a fixed window size to limit the number of packets that can be sent at one time. The window size is specified as an argument to the TCP connection when it is created. Every time an ACK is received, use the send buffer to send any additional packets the window allows. Use a maximum segment size of 1000 bytes. Use sequence numbers that count in terms of bytes. * Use a fixed, one second timer to retransmit segments that are not ACKed in time. When this timer expires, only one segment should be retransmitted. In addition, any data past this segment that has already been transmitted is "forgotten", meaning TCP state is reset and these bytes are not considered to be outstanding. Once the ACK is received for this retransmitted segment, then TCP will "catch up" and send the maximum data allowed by the window. * Keep only one retransmission timer per connection. The rules for this timer are as follows: (1) Every time a segment containing data is sent (including a retransmission), if the timer is not running, start it running so that it will expire after 1 second. (2) When all outstanding data has been acknowledged, turn off the retransmission timer. (3) When an ACK is received that acknowledges new data, restart the retransmission timer so that it will expire after 1 second. When the retransmission timer expires, retransmit the earliest segment that has not been acknowledged and restart the timer so that it will expire after RTO seconds. This is taken from section 5 of [RFC 2988](http://www.ietf.org/rfc/rfc2988.txt). You should implement the following for the **receiver**: * When receiving data, use a receive buffer that can store out-of-order packets. Deliver data to the application only when it is in order. * Send a cumulative ACK for every packet received. The ACK number represents the largest, in-order sequence number that has not yet been received. In other words, the receiver tells the sender the next in-order sequence number it is expecting. Do not implement delayed ACKs. The source code includes an implementation of both a send buffer and a receiver buffer. You should not modify this code. The send buffer implements a basic sliding window algorithm, which is a buffer with three values called *base*, *next*, and *last*. The buffer is a sequence of bytes that can be transmitetd. The *base* of the window is the sequence number of the first byte in the buffer. The *next* value is the sequence number of the next data that has not yet been sent. Everything between the *base* and *next* is data that has been sent and not yet ACKed. The *last* value is the sequence number for the last byte in the buffer. Use the *put()* method to add data to the buffer, and the *get()* method to remove data from the buffer. use the *resend()* method to get data that needs to be resent. Use the *slide()* method when ACKing data that has arrived in order. The receive buffer holds all the data that have been received, indexed by starting sequence number. Data may come in out of order, and the buffer will order them. Data may also be duplicated, and the buffer will remove any duplicated bytes. ## Dynamic Retransmission Timer Once you have basic TCP reliability working, add a dynamic retransmission timer that uses an EWMA for the timer value and for the variance.
